# AGroup DPO 大模型对齐项目方案

> 版本：v1.0
> 适用范围：基于 Qwen2.5-1.5B-Instruct 的保险业务 DPO 对齐训练、分布式训练框架研发、推理加速与评测一体化平台
> 关联文档：[项目需求.md](./项目需求.md)

---

## 第 1 章 项目概述

### 1.1 项目背景

保险业务存在大量与产品条款、理赔规则、健康告知等强相关的问答场景。当前司内已部署"普通 RAG 端"，其检索增强的答案在事实性、可控性、合规口径一致性上仍有提升空间：

- 普通 RAG 端在面对相近问题但不同上下文时，答案表述差异较大，存在"答非所问""过度承诺""遗漏免责条款"等业务风险。
- 单纯依靠提示工程难以稳定控制答案风格与合规边界，需要通过偏好对齐（Preference Alignment）使模型输出更贴合保险业务规范。
- LLaMA-Factory 等开源训练框架提供了 SFT/LoRA/DPO 的成熟流水线，但在大模型分布式训练与多推理后端切换层面仍缺乏面向司内场景的统一封装。

本项目以项目实习为时间窗口，目标是把 LoRA 微调、DPO 对齐、分布式训练（ZeRO3 / FSDP / Megatron / accelerate / deepspeed 多后端）、推理加速（vLLM / xinference 可切换）、自动化评测整合为一套可交付的工程化方案，最终将训练好的 Qwen2.5-1.5B-Instruct 对齐模型与司内普通 RAG 端完成对接。

### 1.2 项目目标

#### 1.2.1 业务目标

| 目标 | 描述 | 衡量方式 |
|------|------|----------|
| G-B1 | 在医疗问答、保险条款问答场景下，模型答案准确率显著优于司内普通 RAG 端 | Accuracy 提升 ≥ 10%（相对值） |
| G-B2 | 答案合规口径统一，对免责条款、健康告知的引用覆盖率 ≥ 95% | 规则化校验脚本 |
| G-B3 | 模型与司内普通 RAG 端完成对接，能在生产推理链路上线 | 端到端连通性测试 |

#### 1.2.2 技术目标

| 目标 | 描述 | 衡量方式 |
|------|------|----------|
| G-T1 | 完成 Qwen2.5-1.5B-Instruct 在 2×A100 上的 ZeRO3 分布式 LoRA 微调 + DPO 对齐训练 | 训练 loss 收敛，模型可 merge 导出 |
| G-T2 | 自研 LoRA+DPO 分布式 Trainer，原生支持 FSDP / Megatron / accelerate / deepspeed 四种后端 | 同一份训练配置可在 4 种后端上跑通 |
| G-T3 | 推理后端可在 vLLM 与 xinference 之间按需切换，切换无需修改业务代码 | 切换耗时 < 5 min，单测覆盖 |
| G-T4 | 完成 Accuracy / BLEU / ROUGE 三类评测指标与推理时间统计的自动化评测流水线 | 评测报告自动产出 JSON + Markdown |

### 1.3 项目范围

**In-Scope**

- 基于 Qwen2.5-1.5B-Instruct 的 LoRA 微调与 DPO 对齐训练流水线
- xx 保险业务的 DPO 数据集生成流水线（含 chosen/rejected 构造）
- 自研 LoRA+DPO 分布式 Trainer（支持 FSDP / Megatron / accelerate / deepspeed）
- 模型 merge 导出（LoRA 权重与基座 Qwen2.5-1.5B-Instruct 合并）
- 推理加速后端 vLLM / xinference 的可切换封装
- Accuracy / BLEU / ROUGE + 推理时间 的评测流水线
- 与司内普通 RAG 端对接的接口契约

**Out-of-Scope**

- 不涉及 RLHF 中的 reward model 训练（仅使用 DPO 离线偏好对齐）
- 不涉及基础模型（Qwen2.5-1.5B-Instruct）的预训练或全参数 SFT
- 不涉及多模态能力
- 不涉及生产级 K8s 部署与高可用架构（项目交付物为离线训练 + 离线评测 + 对接 Demo）

### 1.4 名词术语表

| 术语 | 英文 / 全称 | 释义 |
|------|------------|------|
| DPO | Direct Preference Optimization | 直接偏好优化，基于 chosen/rejected 对的离线对齐算法 |
| LoRA | Low-Rank Adaptation | 低秩适配微调方法，仅训练少量增量参数 |
| SFT | Supervised Fine-Tuning | 监督微调 |
| ZeRO3 | Zero Redundancy Optimizer Stage 3 | DeepSpeed 的优化器状态/梯度/参数三级分片 |
| FSDP | Fully Sharded Data Parallel | PyTorch 原生的全分片数据并行 |
| Megatron | Megatron-LM | NVIDIA 的张量/流水线并行训练框架 |
| accelerate | HuggingFace Accelerate | 轻量分布式训练封装库 |
| deepspeed | DeepSpeed | Microsoft 的分布式训练优化库 |
| vLLM | - | 高吞吐 LLM 推理引擎，基于 PagedAttention |
| xinference | Xorbits Inference | 国产化 LLM 推理服务平台 |
| LLM-as-Judge | - | 用一个大模型作为评判者打分 |
| RAG | Retrieval-Augmented Generation | 检索增强生成 |

### 2.3 约束条件

- **硬件约束**：仅 2×A100-80G 可用于训练，需在 ZeRO3 / FSDP / Megatron 中选择能跨 2 卡工作的策略。
- **数据约束**：DPO 数据来源限定在 xx 保险业务语料（条款、FAQ、历史工单），不允许引入其他业务线数据。
- **合规约束**：所有训练样本需经过 PII 脱敏；模型输出需可追溯到训练数据来源。
- **时间约束**：实习周期内完成从框架熟悉到端到端评测的全链路交付。
- **接口约束**：对接司内普通 RAG 端时，需遵循 RAG 端既有的 HTTP / RPC 接口契约，不得改动 RAG 端代码。

### 2.4 验收标准

| 类别 | 验收项 | 量化阈值 |
|------|--------|----------|
| 训练验收 | LoRA 微调 loss 收敛 | 最终 100 step 窗口内 loss 平均值下降 ≥ 50%（相对初始值） |
| 训练验收 | DPO 对齐 loss 收敛 | chosen-reward - rejected-reward margin ≥ 0.5 |
| 训练验收 | 模型可 merge 导出 | merge 后模型在医疗问答测试集上 perplexity 不高于原模型 1.2 倍 |
| 评测验收 | Accuracy | 医疗问答测试集 ≥ 基准 RAG 端 + 10%（相对值） |
| 评测验收 | BLEU-4 | ≥ 0.30 |
| 评测验收 | ROUGE-L | ≥ 0.45 |
| 性能验收 | 推理时间 | p50 latency ≤ 1.2 s / sample，p95 latency ≤ 2.5 s / sample |
| 框架验收 | 多后端支持 | 同一份 config 至少在 2 种分布式后端（DeepSpeed 与 FSDP）上成功跑通训练 |
| 框架验收 | 推理后端切换 | vLLM ↔ xinference 切换通过单测，切换耗时 < 5 min |

## 第 3 章 技术选型

### 3.1 基础模型选型

**选型结论**：Qwen2.5-1.5B-Instruct

**选型理由**：

| 维度 | Qwen2.5-1.5B-Instruct | 备选：Qwen2.5-7B-Instruct | 备选：Llama-3.2-3B-Instruct |
|------|------------------------|----------------------------|------------------------------|
| 中文能力 | 强（保险业务以中文为主） | 强 | 中（中文弱） |
| 显存占用（LoRA） | ~6 GB | ~24 GB | ~10 GB |
| 2×A100 可行性 | 完全可行 | 需 FSDP/Megatron 严格控制 | 可行 |
| 推理速度 | 快 | 慢 | 中 |
| 与 LLaMA-Factory 兼容性 | 原生支持 | 原生支持 | 原生支持 |

最终选择 1.5B 规模：在 2×A100 上既能在不切 Megatron 的前提下完成 LoRA+DPO，也能保证推理时 vLLM 高吞吐。

### 3.2 训练框架选型

**选型结论**：以 LLaMA-Factory 作为"训练流水线编排层"，自研 Trainer 作为"分布式后端抽象层"。

**职责划分**：

```
┌─────────────────────────────────────────────┐
│             LLaMA-Factory                    │
│  - 数据加载（SFT/DPO 模板）                  │
│  - 模型加载（HF transformers + LoRA 注入）   │
│  - 训练循环（Hugging Face Trainer）          │
│  - LoRA / DPO loss 计算                      │
│  - yaml 配置驱动                             │
└─────────────────────────────────────────────┘
                    ↑ 适配
┌─────────────────────────────────────────────┐
│             自研 Trainer（FR-07）            │
│  - 4 种分布式后端统一接口                    │
│  - 优化器/调度器工厂                         │
│  - ZeRO3 配置生成器                          │
│  - FSDP wrap policy                          │
│  - Megatron 张量并行配置                     │
└─────────────────────────────────────────────┘
```

LLaMA-Factory 负责"训练什么"，自研 Trainer 负责"用什么后端跑"。

### 3.3 分布式训练后端选型

| 后端 | 优势 | 劣势 | 适用场景 | 2×A100 可行性 |
|------|------|------|----------|---------------|
| DeepSpeed (ZeRO3) | 优化器状态/梯度/参数全分片，显存节省显著 | 配置复杂度中等 | 默认主选 | 完全可行 |
| PyTorch FSDP | PyTorch 原生，与 HF Trainer 集成度好 | 早期版本通信开销大 | DeepSpeed 的备选 | 完全可行 |
| Megatron-LM | 张量并行 + 流水线并行，性能极强 | 配置复杂，依赖 NVIDIA 生态 | 模型规模 > 7B 时 | 1.5B 略显重，但需保留接口 |
| HuggingFace accelerate | 上手快，适合小规模实验 | 大规模训练性能不如前三者 | 单卡/小规模调试 | 完全可行 |

**选型决策**：

- **主路径**：DeepSpeed (ZeRO3)，作为项目默认训练后端。
- **备选路径**：FSDP，作为 DeepSpeed 在 PyTorch 原生派系下的等价替代。
- **接口预留**：accelerate 用于单卡调试与 CI 烟雾测试；Megatron 用于后续扩展到 7B+ 模型时的预研接口。

### 3.4 推理加速后端选型

| 后端 | 优势 | 劣势 | 适用场景 | 切换成本 |
|------|------|------|----------|----------|
| vLLM | PagedAttention + continuous batching，吞吐极高 | 对小模型优势略减 | 高吞吐生产推理 | 中（API 略有差异） |
| xinference | 国产化，支持多模型统一管理，部署友好 | 单请求吞吐略低于 vLLM | 国内合规、云上部署 | 中（API 略有差异） |

**选型决策**：两个后端并行支持，业务侧通过配置文件 `infer.backend: vllm | xinference` 切换，无需改动业务代码。

### 3.5 评测工具选型

- **Accuracy**：分类式问答（单选/多选）使用正则抽取选项 + 字符串比对；开放式问答使用 LLM-as-Judge 作为辅助。
- **BLEU**：使用 `nltk.translate.bleu_score`（或 `sacrebleu`）实现 BLEU-1/2/3/4。
- **ROUGE**：使用 `rouge-score`（Google Research 官方实现）计算 ROUGE-1/2/L。
- **推理时间**：使用 `time.perf_counter` 在评测脚本中记录每条样本的首 token 时延与全句时延；统计 p50 / p95 / p99 与总吞吐量。

### 3.6 选型决策总结表

| 决策项 | 结论 | 备选 |
|--------|------|------|
| 基础模型 | Qwen2.5-1.5B-Instruct | Qwen2.5-7B-Instruct（仅在显存允许时启用） |
| 训练框架 | LLaMA-Factory + 自研 Trainer | 纯自研 Trainer（成本过高） |
| 分布式后端（主） | DeepSpeed ZeRO3 | FSDP |
| 分布式后端（备） | FSDP / accelerate | Megatron（仅留接口） |
| 推理后端 | vLLM（默认）+ xinference（备选） | - |
| DPO 算法 | 标准 DPO | IPO / KTO（保留扩展点） |
| 评测指标 | Accuracy + BLEU-4 + ROUGE-L | BLEU-1/2/3、ROUGE-1/2（辅助） |
| 配置驱动 | YAML | JSON / TOML（不支持） |

### 5.3 DPO 对齐训练模块（M-DPO，对应 FR-02）

#### 5.3.1 DPO 原理简述

DPO 损失（Rafailov et al., 2023）：

\[
\mathcal{L}_{\mathrm{DPO}}(\theta) = -\mathbb{E}_{(x,y_w,y_l)\sim D}\left[\log \sigma\left(\beta \log\frac{\pi_\theta(y_w|x)}{\pi_{\mathrm{ref}}(y_w|x)} - \beta \log\frac{\pi_\theta(y_l|x)}{\pi_{\mathrm{ref}}(y_l|x)}\right)\right]
\]

其中 \(y_w\) 为 chosen，\(y_l\) 为 rejected，\(\pi_{\mathrm{ref}}\) 为冻结的参考模型（一般为 SFT 后的模型），\(\beta\) 控制偏离参考模型的强度。

#### 5.3.3 LoRA + DPO 的组合策略

| 策略 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| 策略 1（推荐） | 在 SFT 之后的 LoRA 之上**再叠加一层新的 LoRA**，只训练 DPO LoRA | 显存友好，DPO LoRA 可与 SFT LoRA 解耦 merge | 双 LoRA 需分别管理 |

### 5.4 自研分布式 Trainer 框架（M-TRAINER，对应 FR-07）

#### 5.4.1 设计目标

1. **后端可插拔**：同一份训练配置可在 DeepSpeed / FSDP / Megatron / accelerate 间切换，业务代码不变。
2. **与 LLaMA-Factory 共存**：自研 Trainer 作为 LLaMA-Factory 的"分布式后端适配层"，不替换其训练循环。
3. **配置驱动**：仅通过 yaml 字段 `trainer.distributed_backend: deepspeed | fsdp | megatron | accelerate` 切换。
4. **可观测**：统一暴露 loss / grad_norm / gpu_mem / throughput 指标给 TensorBoard。

#### 5.4.2 核心抽象类（伪代码）

```python
# m_trainer/backends/base.py
from abc import ABC, abstractmethod
from typing import Any, Callable, Optional

class DistributedBackend(ABC):
    """所有分布式后端的统一接口"""

    @abstractmethod
    def init(self, model: torch.nn.Module, optimizer: torch.optim.Optimizer,
             config: "TrainerConfig") -> tuple[torch.nn.Module, torch.optim.Optimizer]:
        """初始化分布式环境，返回 (wrapped_model, wrapped_optimizer)"""

    @abstractmethod
    def wrap_model(self, model: torch.nn.Module) -> torch.nn.Module:
        """对模型应用分片/并行策略"""

    @abstractmethod
    def wrap_optimizer(self, model: torch.nn.Module,
                       optimizer: torch.optim.Optimizer) -> torch.optim.Optimizer:
        """应用 ZeRO / FSDP 优化器包装"""

    @abstractmethod
    def prepare_dataloader(self, dataloader: torch.utils.data.DataLoader
                          ) -> torch.utils.data.DataLoader:
        """分发 sampler / 注入 rank-aware collator"""

    @abstractmethod
    def barrier(self) -> None:
        """跨卡同步"""

    @abstractmethod
    def state_dict(self) -> dict[str, Any]:
        """统一状态字典，支持断点续训"""

    @abstractmethod
    def load_state_dict(self, state: dict[str, Any]) -> None:
        ...


class ShardingStrategy(ABC):
    """分片策略：决定哪些层/参数被分片"""

    @abstractmethod
    def apply(self, model: torch.nn.Module) -> torch.nn.Module: ...


class OptimizerFactory(ABC):
    """优化器工厂：根据后端选择合适的 optimizer 包装"""

    @abstractmethod
    def build(self, params, base_cfg: dict) -> torch.optim.Optimizer: ...
```

#### 5.4.3 四种后端适配器说明

| 后端 | 适配器类 | 关键实现 |
|------|----------|----------|
| DeepSpeed ZeRO3 | `DeepSpeedBackend` | 调用 `deepspeed.initialize(model, optimizer, config=ds_config)`，根据 yaml 生成 `ds_config`（zero_optimization.stage=3, offload_optimizer=cpu 等） |
| PyTorch FSDP | `FSDPBackend` | 使用 `torch.distributed.fsdp.fully_shard`（PyTorch 2.4+）或 `FullyShardedDataParallel`，auto_wrap_policy 基于 `nn.TransformerBlock` size |
| Megatron-LM | `MegatronBackend` | 注入 tensor_model_parallel_size 与 pipeline_model_parallel_size；1.5B 暂不启用 TP，仅保留接口 |
| accelerate | `AccelerateBackend` | 使用 `accelerator.prepare(model, optimizer, dataloader)`；用于单机多卡调试与 CI |

#### 5.4.4 切换机制（工厂模式 + 配置驱动）

```python
# m_trainer/factory.py
BACKEND_REGISTRY = {
    "deepspeed": "m_trainer.backends.deepspeed:DeepSpeedBackend",
    "fsdp":      "m_trainer.backends.fsdp:FSDPBackend",
    "megatron":  "m_trainer.backends.megatron:MegatronBackend",
    "accelerate":"m_trainer.backends.accelerate:AccelerateBackend",
}

def build_backend(cfg: "TrainerConfig") -> DistributedBackend:
    name = cfg.distributed_backend
    if name not in BACKEND_REGISTRY:
        raise ValueError(f"unsupported backend: {name}, "
                         f"choose from {list(BACKEND_REGISTRY)}")
    module_path, cls_name = BACKEND_REGISTRY[name].split(":")
    cls = importlib.import_module(module_path).__dict__[cls_name]
    return cls()
```

### 5.5 模型 Merge 导出模块（M-MERGE，对应 FR-06）

#### 5.5.1 LoRA 与基座合并算法

合并公式（PEFT 库 `merge_and_unload` 实现）：

\[
W_{\mathrm{merged}} = W_{\mathrm{base}} + \frac{\alpha}{r} \cdot B \cdot A
\]

其中 \(W_{\mathrm{base}}\) 为基座 Qwen2.5-1.5B-Instruct 的原权重，\(A \in \mathbb{R}^{r \times d_{\mathrm{in}}}\)，\(B \in \mathbb{R}^{d_{\mathrm{out}} \times r}\) 为 LoRA 矩阵，\(\alpha\) 为缩放系数，\(r\) 为 rank。

#### 5.5.2 导出格式

| 格式 | 适用场景 | 是否默认 |
|------|----------|----------|
| HuggingFace safetensors（单文件夹） | 直接被 vLLM / xinference / HF transformers 加载 | ✅ 默认 |
| 合并后单文件 .bin（兼容老格式） | 兼容部分老推理框架 | 备选 |
| LoRA adapter 单独保留 | 仅保留增量参数，便于回滚 | 与 merge 产物并存 |

### 5.6 推理加速后端（M-INFER，对应 FR-08 上半）

#### 5.6.1 统一推理接口设计

```python
# m_infer/base.py
from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import Optional

@dataclass
class InferRequest:
    prompt: str
    max_new_tokens: int = 512
    temperature: float = 0.7
    top_p: float = 0.9
    stop: Optional[list[str]] = None
    request_id: Optional[str] = None

@dataclass
class InferResponse:
    text: str
    prompt_tokens: int
    generated_tokens: int
    latency_ms: float        # 首 token 时延
    total_latency_ms: float  # 全句时延
    request_id: Optional[str] = None


class InferBackend(ABC):
    """推理后端抽象类"""

    @abstractmethod
    def load(self, model_path: str, **kwargs) -> None: ...

    @abstractmethod
    def infer(self, req: InferRequest) -> InferResponse: ...

    @abstractmethod
    def batch_infer(self, reqs: list[InferRequest]) -> list[InferResponse]: ...

    @abstractmethod
    def shutdown(self) -> None: ...


class InferBackendFactory:
    _registry = {
        "vllm":      "m_infer.vllm_backend:VLLMBackend",
        "xinference": "m_infer.xinference_backend:XinferenceBackend",
    }

    @classmethod
    def build(cls, name: str, model_path: str, **kwargs) -> InferBackend:
        if name not in cls._registry:
            raise ValueError(f"unsupported infer backend: {name}")
        module_path, cls_name = cls._registry[name].split(":")
        backend_cls = importlib.import_module(module_path).__dict__[cls_name]
        backend = backend_cls()
        backend.load(model_path, **kwargs)
        return backend
```

#### 5.6.2 vLLM 后端实现要点

```python
# m_infer/vllm_backend.py
from vllm import LLM, SamplingParams

class VLLMBackend(InferBackend):
    def load(self, model_path: str, tensor_parallel_size: int = 1, **kw):
        self._llm = LLM(
            model=model_path,
            tensor_parallel_size=tensor_parallel_size,
            dtype="bfloat16",
            gpu_memory_utilization=0.85,
            max_model_len=2048,
            enforce_eager=False,    # 启用 cuda graph
        )

    def infer(self, req: InferRequest) -> InferResponse:
        params = SamplingParams(
            temperature=req.temperature,
            top_p=req.top_p,
            max_tokens=req.max_new_tokens,
            stop=req.stop,
        )
        t0 = time.perf_counter()
        outputs = self._llm.generate([req.prompt], params, use_tqdm=False)
        total_ms = (time.perf_counter() - t0) * 1000
        # vLLM 内部已统计首 token 时延，outputs[0].metrics.first_token_time 可获取
        out = outputs[0]
        return InferResponse(
            text=out.outputs[0].text,
            prompt_tokens=len(out.prompt_token_ids),
            generated_tokens=len(out.outputs[0].token_ids),
            latency_ms=out.metrics.first_token_time if out.metrics else 0.0,
            total_latency_ms=total_ms,
            request_id=req.request_id,
        )
```

**vLLM 关键技术**：

- PagedAttention：分页式 KV cache 管理，显存利用率提升 4× 以上。
- Continuous batching：请求级调度，不等整批完成即可插入新请求。
- Chunked prefill：长 prompt 分块 prefill，避免阻塞短请求 decode。

#### 5.6.3 xinference 后端实现要点

```python
# m_infer/xinference_backend.py
import requests

class XinferenceBackend(InferBackend):
    """xinference 通过 HTTP / gRPC 与本地/远程 xinference 服务通信"""

    def load(self, model_path: str, server_endpoint: str = "http://127.0.0.1:9997",
             model_uid: Optional[str] = None, **kw):
        # 1. 注册模型（若尚未注册）
        if model_uid is None:
            resp = requests.post(f"{server_endpoint}/v1/models",
                                 json={"model_name": "qwen2_5_insurance",
                                       "model_path": model_path,
                                       "model_type": "LLM"})
            model_uid = resp.json()["model_uid"]
        self._endpoint = server_endpoint
        self._model_uid = model_uid

    def infer(self, req: InferRequest) -> InferResponse:
        t0 = time.perf_counter()
        resp = requests.post(
            f"{self._endpoint}/v1/completions",
            json={
                "model": self._model_uid,
                "prompt": req.prompt,
                "max_tokens": req.max_new_tokens,
                "temperature": req.temperature,
                "top_p": req.top_p,
                "stop": req.stop,
            },
        )
        total_ms = (time.perf_counter() - t0) * 1000
        data = resp.json()
        return InferResponse(
            text=data["choices"][0]["text"],
            prompt_tokens=data["usage"]["prompt_tokens"],
            generated_tokens=data["usage"]["completion_tokens"],
            latency_ms=data.get("first_token_latency_ms", 0.0),
            total_latency_ms=total_ms,
            request_id=req.request_id,
        )
```

**xinference 关键技术**：

- 统一模型注册中心（xgboost / pytorch / LLM / embedding 一站式）。
- 内置 vLLM / transformers / llama.cpp 等多种推理后端，可按需选。
- 支持模型量化（GPTQ / AWQ / BNB），对国产 GPU 适配较好。

#### 5.7.3 推理时间统计方法

```python
# m_eval/latency.py
import time
import numpy as np

@dataclass
class LatencyStat:
    p50_first_token_ms: float
    p95_first_token_ms: float
    p99_first_token_ms: float
    p50_total_ms: float
    p95_total_ms: float
    p99_total_ms: float
    throughput_samples_per_s: float

def aggregate_latency(records: list[InferResponse]) -> LatencyStat:
    first = np.array([r.latency_ms for r in records])
    total = np.array([r.total_latency_ms for r in records])
    elapsed = sum(r.total_latency_ms for r in records) / 1000.0
    return LatencyStat(
        p50_first_token_ms=float(np.percentile(first, 50)),
        p95_first_token_ms=float(np.percentile(first, 95)),
        p99_first_token_ms=float(np.percentile(first, 99)),
        p50_total_ms=float(np.percentile(total, 50)),
        p95_total_ms=float(np.percentile(total, 95)),
        p99_total_ms=float(np.percentile(total, 99)),
        throughput_samples_per_s=len(records) / elapsed,
    )
```

### 6.3 数据质量保证

| 质量维度 | 检查规则 | 不通过处理 |
|----------|----------|------------|
| 长度 | prompt 长度 ∈ [5, 1024]；chosen/rejected ∈ [10, 2048] | 丢弃或人工复核 |
| 敏感词 | 包含身份证/手机号/银行卡/邮箱原始格式则不通过 | 强制 PII 脱敏 |
| 重复 | 同一 prompt 不允许多次出现（按 hash 去重） | 保留最新版本 |
| 必引条款 | 涉及"赔付/等待期/免赔/告知"等关键词的 chosen 必须包含"条款/保单/合同"之一 | 回流到 PairBuilder 重做 |
| chosen≠rejected | 字符级相似度 < 0.95 | 丢弃 |
| PII | 100% 通过正则+词典双重脱敏 | 整条丢弃 |

**规则校验伪代码**：

```python
# m_data/validator.py
import re

PII_PATTERNS = [
    re.compile(r"\d{17}[\dXx]"),                       # 身份证
    re.compile(r"1[3-9]\d{9}"),                         # 手机号
    re.compile(r"\d{16,19}"),                           # 银行卡
    re.compile(r"[\w.+-]+@[\w-]+\.[\w.-]+"),            # 邮箱
]

REQUIRED_TERMS_FOR_CLAIMS = ["条款", "保单", "合同", "法"]

def validate(sample: dict) -> tuple[bool, str]:
    if not (5 <= len(sample["prompt"]) <= 1024):
        return False, "prompt length out of range"
    for field in ("prompt", "chosen", "rejected"):
        for pat in PII_PATTERNS:
            if pat.search(sample[field]):
                return False, f"PII detected in {field}"
    if any(kw in sample["chosen"] for kw in ["赔付", "等待期", "免赔", "告知"]):
        if not any(t in sample["chosen"] for t in REQUIRED_TERMS_FOR_CLAIMS):
            return False, "missing required policy reference"
    if similarity(sample["chosen"], sample["rejected"]) > 0.95:
        return False, "chosen and rejected too similar"
    return True, "ok"
```

| rouge-score | 0.1.2+ | ROUGE |

### 8.3 Docker 镜像设计

采用"基础镜像 + 后端 layer"的分层设计，避免一个超大镜像：

```
# 基础镜像
copaw-dpo-base:latest
  - python3.10 + cuda12.4 + cudnn8.9 + pytorch2.4 + transformers + LLaMA-Factory

# 后端 layer（在 base 之上叠加）
copaw-dpo-deepspeed:latest   # + deepspeed + nvtx
copaw-dpo-fsdp:latest        # + pytorch fsdp + nvtx
copaw-dpo-megatron:latest    # + megatron-lm + apex
copaw-dpo-accelerate:latest  # + accelerate

# 推理 layer
copaw-dpo-infer-vllm:latest   # + vllm
copaw-dpo-infer-xinfer:latest # + xinference + docker-compose

# 评测 layer
copaw-dpo-eval:latest         # + sacrebleu + rouge-score + judge 模型
```

启动命令示例：

```bash
# 训练（DeepSpeed）
docker run --gpus all --rm -it \
    -v /shared:/shared \
    copaw-dpo-deepspeed:latest \
    copaw-dpo train-dpo --config configs/train_dpo_qwen2_5_1_5b_insurance.yaml

# 推理（vLLM）
docker run --gpus all --rm -it -p 8080:8080 \
    -v /shared/models:/models \
    copaw-dpo-infer-vllm:latest \
    copaw-dpo infer --config configs/infer.yaml --host 0.0.0.0 --port 8080
```

### 9.3 各阶段说明

> 本节按统一结构（阶段定位 / 目标 / 核心任务 / 交付物 / 验收标准 / 依赖关系）描述每个阶段，详细阶段文档可后续拆分为独立的 `M0X_xxx.md` 文件并放入 `./项目分阶段方案/` 子目录。

#### M01 基础设施与环境准备

- **阶段定位**：所有后续阶段的硬前置。
- **目标**：在 2×A100-80G 训练节点上完成 LLaMA-Factory、DeepSpeed、vLLM、xinference 等依赖的部署与烟雾测试。
- **核心任务**：
  1. 拉取并构建 Docker 镜像（base + 4 种后端 layer）。
  2. 验证 2 卡 NVLink 互联。
  3. 跑通 LLaMA-Factory 单卡 SFT 烟雾测试（用 Qwen2.5-1.5B-Instruct + 极小数据集）。
  4. 跑通 vLLM 与 xinference 的单卡推理烟雾测试。
- **交付物**：
  - `deploy/Dockerfile.base` + 4 个 layer Dockerfile
  - `deploy/docker-compose.yml`（推理 / 训练 / 评测 三套）
  - 烟雾测试通过的日志
- **验收标准**：
  - LLaMA-Factory 单卡 SFT 1 epoch 成功，loss 下降
  - vLLM 与 xinference 均能加载 base 模型并返回正常文本
  - 2 卡 torchrun hello-world 通信成功
- **依赖关系**：无上游。

#### M02 DPO 数据集生成流水线

- **阶段定位**：训练阶段的输入供给。
- **目标**：交付 `dpo_train_v1.2.jsonl`，覆盖 ≥ 5000 条 chosen/rejected 对。
- **核心任务**：
  1. 实现三类数据源（条款 / FAQ / 工单）的采集器。
  2. 实现三类构造策略（规则硬负例 / LLM-as-Judge / 检索差异）。
  3. 实现 PII 脱敏与规范化流程。
  4. 实现规则校验（必引条款、chosen≠rejected 等）。
  5. 与司内普通 RAG 端对接获取 baseline 答案。
- **交付物**：
  - `m_data/` 模块完整代码
  - `data/insurance/dpo_train_v1.2.jsonl`（≥ 5000 条）
  - 数据质量报告（通过率 / 各类 source 占比 / PII 命中率 0）
  - 流水线运行说明 README
- **验收标准**：
  - 通过率 ≥ 85%
  - PII 命中率 = 0
  - 三类 source 样本分布合理（无单一 source 占比 > 70%）
- **依赖关系**：依赖 M01。

#### M03 LoRA 微调与 DPO 对齐训练

- **阶段定位**：核心训练阶段，对齐模型的产出方。
- **目标**：在 2×A100 上完成 LoRA SFT（可选）+ DPO 对齐训练，产出可 merge 的 LoRA checkpoint。
- **核心任务**：
  1. 编写 LLaMA-Factory yaml 配置（SFT / DPO 两份）。
  2. 完成 LoRA SFT 训练（可选）。
  3. 完成 DPO 对齐训练。
  4. 在 DeepSpeed ZeRO3 主路径上跑通。
  5. 在 FSDP 备选路径上至少跑通一次烟雾测试。
  6. 完成 merge 导出（依赖 M04 的统一 state_dict 接口）。
- **交付物**：
  - `configs/train_lora_qwen2_5_1_5b_insurance.yaml`
  - `configs/train_dpo_qwen2_5_1_5b_insurance.yaml`
  - `saves/qwen2_5_1_5b/insurance_dpo_v1.2/lora/`（最终 LoRA 权重）
  - 训练日志（loss 曲线截图 + TensorBoard 链接）
- **验收标准**：
  - DPO chosen-reward - rejected-reward margin ≥ 0.5
  - 同一份 yaml 在 DeepSpeed 与 FSDP 上均可启动训练
- **依赖关系**：依赖 M01、M02；与 M04 强耦合（需 M04 提供统一后端抽象）。

#### M04 自研分布式 Trainer 与模型导出

- **阶段定位**：训练框架层，提升后续扩展能力。
- **目标**：交付自研 `m_trainer` 模块，原生支持 DeepSpeed / FSDP / Megatron / accelerate 四种后端，与 LLaMA-Factory 协作无冲突。
- **核心任务**：
  1. 设计 `DistributedBackend` / `ShardingStrategy` / `OptimizerFactory` 抽象接口。
  2. 实现 4 个后端适配器。
  3. 实现工厂 + 注册机制。
  4. 与 LLaMA-Factory 集成：让 LLaMA-Factory 的训练循环能消费自研 backend 的 wrapped_model/wrapped_optimizer。
  5. 实现统一 state_dict 的 checkpoint 保存/加载。
  6. 单测覆盖：每个后端至少 1 个端到端训练冒烟用例。
- **交付物**：
  - `m_trainer/` 模块完整代码 + 单测
  - `docs/项目分阶段方案/M04_自研Trainer.md`（如后续拆分）
  - 4 种后端的冒烟测试日志
- **验收标准**：
  - 同一份 yaml + 同一份数据集在 4 种后端上均能启动训练
  - 单测覆盖率 ≥ 60%
  - checkpoint 可在 4 种后端间互换加载
- **依赖关系**：依赖 M01；与 M03 并行（接口先行，实现同步）。

#### M05 推理加速与评测

- **阶段定位**：交付与验收阶段。
- **目标**：交付可切换的 vLLM/xinference 推理服务，并产出 Accuracy/BLEU/ROUGE + 推理时间 的评测报告。
- **核心任务**：
  1. 实现 `InferBackend` 抽象类 + `VLLMBackend` + `XinferenceBackend`。
  2. 实现评测指标计算（accuracy / bleu_4 / rouge_l）。
  3. 实现推理时间统计（p50/p95/p99 + throughput）。
  4. 实现评测报告产出（JSON + Markdown）。
  5. 与司内普通 RAG 端对接：定义 HTTP 协议，跑通端到端连通性测试。
  6. baseline 对比：对比"未做 DPO 的基线模型"与"DPO 模型"在医疗问答测试集上的差异。
- **交付物**：
  - `m_infer/` 模块完整代码 + 单测
  - `m_eval/` 模块完整代码 + 单测
  - `reports/eval_report_dpo_v1.2.json` 与 `.md`
  - 与司内 RAG 端对接的 Demo 视频/截图
- **验收标准**：
  - vLLM ↔ xinference 切换通过单测，切换耗时 < 5 min
  - 评测报告完整，含全部 4 项指标
  - 与司内 RAG 端端到端连通性测试通过
  - DPO 模型 Accuracy / BLEU-4 / ROUGE-L 全部超过基线（具体阈值见 2.4 节）
- **依赖关系**：依赖 M03、M04。
